## 0. 开发范式（决定一切）
- 负责人**不写代码**，是"AI 的架构师 + 质检员"。你（AI）产出，他验收。
- 六铁律：① 一切可验证化（交付要给能点的 demo/录屏/数据，不是"我写完了"）② 模块可独立理解/修改/验证 ③ 接口先行 ④ 配置与逻辑分离（数据进表，他自己改）⑤ 小步快跑（一次一个可验证小块）⑥ 记忆锚点（每次开工先读本文件 + DECISION_LOG）。

## 1. 模块与职责（一个文件夹=一个模块，边界清晰）
- M1 forging 锻造交互：手势→改材料状态。不算最终数值、不碰故事、不渲染。
- M2 weapon WeaponData：唯一真相源，纯数据结构。
- M3 evaluation 数值评估：材料状态→6维+总评分。**Demo 用简化取舍公式；未来可升级物理模拟，对外仍只出 6 维，升级只改本模块内部、接口不变**。
- M4 adventure 冒险判定：读 6 维 + 判定节点表→分支结果。不关心怎么锻的。
- M5 story 故事呈现：判定结果 + 文案表→PPT 式故事。只负责"讲"。
- M6 render 渲染：读 WeaponData→画 2.5D 外观/光效。**禁止反向写任何数值/状态（只读）**。
- M7 scene 场景导航：2.5D 铁匠铺中枢 + 工具入口。
- M8 save_share 存档/分享：存读进度 + 二维码（传 6 维参数）。
- M9 data 数据配置：材料/工具/判定节点/故事文案/图纸表（负责人直接改，纯数据）。
- M10 ui 界面：数值条/对话框等，不含业务逻辑。
- core：事件总线/服务定位器等基础设施。

## 2. 数据流铁律（绝不违反）
```
玩家手势 → M1 → 写 → M2(WeaponData) ← M3 写回6维
                        │ 只读
              ┌─────────┼──────────┐
          M6渲染    M4冒险判定    M8存档分享   （全部只读 M2）
                        │ 读 M9判定节点表 → M5故事(读M9文案表)
```
- 只有 **M1 和 M3 能写 M2**，其他一律只读。
- **M6 渲染禁止反向写数值**。
- M4/M5 不准直接读 M1（判定只认最终数值）。
- 跨模块通信走**事件总线**，不直接调用别人内部函数。

## 3. 接口契约（WeaponData，全项目普通话，改它须记 DECISION_LOG 并通知全模块）
```ts
WeaponData = {
  shape: { controlPoints: number[], thickness: number[] },  // 外形 M1写 M6读
  material: string,                                          // 材质ID M1写
  process: string[],                                         // 工艺标签 M1写
  stats: { edge, hard, tough, weight, look, balance },       // 6维0-10 M3写 M4/M6读
  overall: number,                                           // 总评分 M3写 M6/M8读
  flaws: string[],                                           // 裂纹等 M1写 M4/M5读
}
```

## 4. 数值设计要点（Demo 期）
- 6 维**不是独立能力值**，是三对取舍：硬度↔韧性 / 锋利↔耐用 / 轻↔结实。无全满神刀。
- 平衡、外观 = 综合工艺奖励，不参与对立。
- 耐久由 硬度+韧性 派生，不单开维度。
- 特殊功能（淬毒/引火等）走 process 工艺标签，不塞进 6 维。
- 总评分不参与冒险判定（判定逐维判）。

## 5. Git 习惯
- 每个模块/小步**验收通过后立即 commit + push**（一个能跑的检查点=一次提交）。
- 提交信息：`[模块] 动作`，如 `[M4] 冒险判定跑通，验收通过`。
- 开工前先 `git status`；改坏了修不回来→回退到上一个提交。

## 6. 其他习惯
一次一个模块 / 改动前先说计划待确认 / 删文件前先问 / 数字文案全进 data 表 / 每模块留可独立点的测试入口 / 报错用人话说清"哪里·为什么·打算怎么办" / 锁引擎与库版本 / 每步更新 PROGRESS.md。

## 7. 施工顺序
S0 环境 → S1 数据+测试料 → **S2 数值→故事(先验证卖点二)** → S3 锻造产出 → S4 画面串通 → S5 磨手感(难点) → S6 社交闭环 → S7 内容+美术。详见全案第四部分。
