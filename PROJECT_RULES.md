## 0. 开发范式（决定一切）
- 负责人**不写代码**，是"AI 的架构师 + 质检员"。你（AI）产出，他验收。
- 六铁律：① 一切可验证化（交付要给能点的 demo/录屏/数据，不是"我写完了"）② 模块可独立理解/修改/验证 ③ 接口先行 ④ 配置与逻辑分离（数据进表，他自己改）⑤ 小步快跑（一次一个可验证小块）⑥ 记忆锚点（每次开工先读本文件 + DECISION_LOG）。

## 1. 模块与职责（一个文件夹=一个模块，边界清晰）
- M1 forging 锻造交互：手势→改材料状态。不算最终数值、不碰故事、不渲染。
- M2 weapon WeaponData：唯一真相源，纯数据结构。
- M3 evaluation 数值评估：材料状态→6维+总评分。**Demo 用简化取舍公式；未来可升级物理模拟，对外仍只出 6 维，升级只改本模块内部、接口不变**。
- M4 adventure 冒险判定：读 6 维 + 判定节点表→分支结果。不关心怎么锻的。
- M5 story 故事呈现：判定结果 + 文案表→PPT 式故事。只负责"讲"。
- M6 render 渲染：读 WeaponData→画 2.5D 外观/光效。**禁止反向写任何数值/状态（只读）**。
- M7 scene 场景导航：2.5D 铁匠铺中枢 + 工具入口。
- M8 save_share 存档/分享：存读进度 + 二维码（传 6 维参数）。
- M9 data 数据配置：材料/工具/判定节点/故事文案/图纸表（负责人直接改，纯数据）。
- M10 ui 界面：数值条/对话框等，不含业务逻辑。
- core：事件总线/服务定位器等基础设施。

## 2. 数据流铁律（绝不违反）
```
玩家手势 → M1 → 写 → M2(WeaponData) ← M3 写回6维
                        │ 只读
              ┌─────────┼──────────┐
          M6渲染    M4冒险判定    M8存档分享   （全部只读 M2）
                        │ 读 M9判定节点表 → M5故事(读M9文案表)
```
- 只有 **M1 和 M3 能写 M2**，其他一律只读。
- **M6 渲染禁止反向写数值**。
- M4/M5 不准直接读 M1（判定只认最终数值）。
- 跨模块通信走**事件总线**，不直接调用别人内部函数。

## 3. 接口契约（WeaponData，全项目普通话，改它须记 DECISION_LOG 并通知全模块）
```ts
WeaponData = {
  shape: { baseForm: string, controlPoints: number[], thickness: number[], surface?: object }, // 外形=锻造实时生成 M1写 M6读；baseForm=刀/剑/斧基础形制(第二层贴图对齐锚点)
  material: string,                                          // 材质ID M1写
  process: string[],                                         // 工艺标签(折叠/分层/淬火介质等) M1写；M6第二层据此判定贴图
  stats: { edge, hard, tough, weight, look, balance },       // 6维0-10 M3写 M4/M6读
  overall: number,                                           // 总评分 M3写 M6/M8读
  flaws: string[],                                           // 裂纹等 M1写 M4/M5读
}
```
> **武器外观=玩家锻造产物，非预制部件拼装**（详见 docs 全案 §3.6 物理外观两层模型）。M6 渲染分两层：① 读 shape/surface 实时生成形状/温度色/光泽；② 读 process+物理量，达标则叠加对应分级工艺贴图(大马士革等)。baseForm 是第二层贴图对齐的骨架。

## 4. 数值设计要点（Demo 期）
- 6 维**不是独立能力值**，是三对取舍：硬度↔韧性 / 锋利↔耐用 / 轻↔结实。无全满神刀。
- 平衡、外观 = 综合工艺奖励，不参与对立。
- 耐久由 硬度+韧性 派生，不单开维度。
- 特殊功能（淬毒/引火等）走 process 工艺标签，不塞进 6 维。
- 总评分不参与冒险判定（判定逐维判）。

## 5. Git 规范（轻量分支流，方案 A，本项目铁律）
- **main 永远稳定可跑**：只有"验收通过"的版本能进 main。禁止在 main 上直接开发。
- **一步一分支**：每个模块/施工步在独立功能分支上做，命名 `feat/<步骤>-<模块>`，如 `feat/S1-data`、`feat/S2-story`。
- **提交粒度**：一个可验证的小改动 = 一次 commit；信息格式 `[模块] 动作`，如 `[M4] 冒险判定跑通，验收通过`。
- **合并前提**：该分支的模块"能独立跑 / 验收通过"才 `merge` 回 main；合并后更新 PROGRESS.md 并（有远程时）push。
- **一分支一模块** = 架构"一次只动一个模块"在流程上的对应，天然防跨模块污染。
- 开工前先 `git status` + 确认在正确分支；改坏了修不回来 → 回退到上一个提交。
- 有 GitHub 远程后：每次合并到 main 都 push，异地备份（换设备接力的前提）。
- **不采用完整 Git Flow**（develop/release/hotfix）——单人过度工程，已否决。

## 7. 锻造系统要点（详见全案 §2 + docs/锻造工艺研究报告.md）
- 工序抽象：选料→加热→锻打→(正火)→淬火→**回火**→研磨→装配；折叠/大马士革=扩展。
- **回火**=玩家拖温度条主动定"硬vs韧"取舍（温度越高越韧越软）。**研磨能改形+定刃角**，非只开刃。
- **准则**："拟真↔游戏"是可滑动旋钮(原型迭代找平衡)；"能离散就不连续"(旋转固定90°)。
- **唯一真技术壁垒**=锻打/研磨实时形变，用"分段缩放"跨越，其余纯逻辑。**手感70%靠反馈(火花/音效/数字)，非形变精度。**
- 判定=宽容的"固定平衡点"(时机条绿区)，可迭代微调，**非动态难度DDA**。
