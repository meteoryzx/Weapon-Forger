# PROJECT_RULES · 每次开工必读

> 平台：PC 键鼠 H5 Demo；引擎：Cocos Creator 3.8.8；语言：TypeScript。冲突时，以本文件和 `DECISION_LOG.md` 最新“替代决策”为准。

## 1. 完成标准

负责人不写代码，按玩家行为验收。一个切片只有同时满足以下条件才算完成：

- 自动测试、类型检查和数据校验通过。
- 适用时有可打开的 Cocos/H5 入口和清楚的复现步骤。
- Codex 完成一次独立 diff review。
- 作者亲手试玩并确认玩家行为。
- `CURRENT_TASK.md`、PR 证据和决策记录已更新。

## 2. 模块边界

| 模块 | 负责 | 禁止 |
|---|---|---|
| M1 forging | 管理锻造会话，把合法输入变成 `ForgeOperation`，推进纯模拟 | 不计算六维，不写故事，不依赖具体 UI |
| M2 weapon | 定义最终不可变 `WeaponData` 和公共数据结构 | 不放业务计算 |
| M3 evaluation | 从最终 `ForgeState` 推导六维、工艺完成度、特征和缺陷 | 不接受“按钮加属性”，不修改施工状态 |
| M4 agent_world | NPC 决策、世界状态和确定性冒险规则 | 不读施工过程，不编造武器事实 |
| M5 story | 把已发生的规则事件转成条件文案 | 不决定模拟结果 |
| M6 render | 读取 `ForgeSnapshot`/`WeaponData` 渲染形变、温度和反馈 | 不反写模拟状态或数值 |
| M7 scene | 场景与流程导航 | 不承载物理或故事规则 |
| M8 save_share | 存档、回放、分享数据 | 不改变结果 |
| M9 data | 材料、工具、参数、规则和文案 | 不藏运行逻辑 |
| M10 ui | 输入适配和界面显示 | 不承担业务判定 |

纯模拟、评估和代理世界不能 import Cocos。输入适配器可替换，键鼠只是首版入口。

## 3. 数据流

```text
键鼠输入 -> M10 输入适配 -> M1 创建 ForgeOperation -> 纯模拟更新 ForgeState
                                                    |
                                                    +-> ForgeSnapshot -> M6 只读渲染

锻造完成 -> M3 只读评估 ForgeState -> 不可变 WeaponData
                                      -> M4 规则代理世界 -> M5 故事呈现
                                      -> M8 存档/分享
```

- `ForgeState` 是施工期唯一真相，包含 `WorkpieceState[]`、`BladeSection[]`、`JointState[]` 和操作历史。
- `ForgeSnapshot` 是给画面的只读、可丢弃投影；画面不能成为物理真相。
- `WeaponData` 只在完成锻造时产生，故事只读它，不读锤击日志。
- 操作只改变材料状态；六维只能由最终状态推导。

## 4. 模拟规则

- 首版是最小规则模拟，不是软体物理：温度、塑性、截面几何、近似体积、应力、完整性和连接。
- 刀坯使用 8-10 个截面；锤击影响当前段和邻近段，截面共同生成连续外观。
- A/D 每次旋转 90°，四面受击必须正确交换宽度/厚度表现。
- 相同初始状态、操作序列、参数版本和种子必须得到相同结果。
- 裂纹由状态超过确定阈值产生，不掷随机概率；随机仅用于已显式带种子的世界内容。
- 规则参数进数据表。每条规则都要有标准样本，能解释“认真锻造、乱敲、低温重击、单侧猛敲”的差别。
- 两天内动态网格无法稳定达到验收，就改用连接式 2D ribbon；模拟接口不变。

## 5. 数值与故事

- 隐藏六维：锋利、硬度、韧性、重量、平衡、外观。它们是故事接口，不是六个独立加点槽。
- `craftsmanship` 只表示工艺完成度，不是万能战力；不保留误导性的 `overall` 总战力。
- 重量是用途参数，不是越轻或越重越好。硬/韧、薄刃/耐用等取舍必须从同一物理状态自然产生。
- `traits`/`flaws` 结构化保存薄刃、前端过重、弯曲、裂纹、连接不良等事实。
- 自由成品不需要 `baseForm` 分类才合法。用途由几何、能力和缺陷决定。
- 故事是性能的唯一明确反馈；锻造界面不显示六维，也不显示“优秀/失败”总结。
- 同一 `WeaponData`、世界初态和种子必须产生同一冒险。故事文本只能描述规则事件已经证明的事实。

## 6. Git 与协作

- `main` 始终可运行；所有修改经短期 branch 和 PR，禁止直接开发。
- 一个 PR 可以跨模块，但只能交付一条玩家可验证行为。禁止 M1/M3/M6 各自长期开发后集中拼接。
- 第一个检查点即 push 并建立 Draft PR；半成品只存在于分支和 Draft PR。
- 每个可描述检查点 commit；验收后 squash merge，使 `main` 一条切片一个正式提交。
- 公共接口先写契约测试，再实现调用方。破坏性接口变更必须记录 `DECISION_LOG.md`。
- 日常禁止 `git reset --hard` 和 force push。出问题先建 `rescue/<日期>-<说明>`，已合入历史用 `git revert`。
- 两台设备不同时开发；离开设备前关闭 Cocos、运行检查、更新 `CURRENT_TASK.md`、commit、push、核对远端并保持工作区干净。

## 7. 项目文件

- 必须提交：`assets/`、全部 `.meta`、`settings/`、Cocos `package.json`、锁文件、源码和数据。
- 禁止提交：`library/`、`temp/`、`local/`、`build/`、依赖目录和本机缓存。
- PSD、WAV、BLEND 等大型源资产走 Git LFS；构建压缩包和视频走 GitHub Release。
- 数字、材料参数、委托规则和玩家文案进入数据文件，不散落硬编码。

## 8. 提交前检查

- [ ] 当前 branch、Issue 和 PR 正确，工作区每个变化都已理解。
- [ ] 只实现本切片声明的玩家行为，没有顺手扩范围。
- [ ] 模拟无 Cocos 依赖，渲染无反向写状态，故事只读最终数据。
- [ ] 公共接口、确定性和极端样本有测试。
- [ ] 参数/文案在数据表，Cocos `.meta` 完整，生成目录未进入 Git。
- [ ] 自动检查已实际运行；未运行或失败的项目被明确记录。
- [ ] 作者有无需读代码即可完成的试玩步骤。
