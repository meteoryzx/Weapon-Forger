# START_HERE · 接手《打了个铁》的第一份指南

> 本文件是冷启动入口。它记录项目目标、当前路线和绝不能突破的边界；具体进度以 `CURRENT_TASK.md` 和 `PROGRESS.md` 为准。

## 1. 项目是什么

《打了个铁》是一个面向系统策划秋招作品集的浏览器 H5 Demo。玩家像《锻刀大会》一样自由完成选料、加热、锻打、淬火、研磨、回火和装配；游戏用一套可解释、可扩展的最小物理规则回应玩家操作，并从成品状态推导隐藏六维和结构化缺陷。NPC 拿着武器自主冒险，故事后果是玩家得到的明确性能反馈。

两个卖点必须组成一个闭环：

1. 玩家真的通过操作改变材料、外形和缺陷，而不是按配方换属性。
2. 故事真的由最终武器数据和世界状态产生，而不是用文案假装有因果。

## 2. 接手顺序

1. 读 `AGENTS.md` 和 `CURRENT_TASK.md`。
2. 读本文件。
3. 依次读 `PROGRESS.md`、`PROJECT_RULES.md`、`DECISION_LOG.md`。

5. 开工前报告当前 branch、工作区状态、准备改变的玩家行为和验收证据。

## 4. 绝不能做

- 不直接在 `main` 开发，不把未验收半成品合入 `main`，不 force push。
- 不以“一个模块一个长期分支”并行开发后集中拼装；每个 PR 只完成一条玩家可验证行为。
- 不用 `git reset --hard` 处理日常问题；优先 rescue 分支和 `git revert`。
- 不做软体/有限元等高成本真实物理；要做的是确定性、可解释、可扩展的最小规则模拟。
- 不把锻造操作写成“硬度 +2”；操作先改变温度、几何、应力、完整性等物理状态，M3 再评估。
- 不让渲染或故事反向修改模拟；Cocos 只提交操作并读取只读快照，故事只读最终 `WeaponData`。
- 不强迫自由成品归类为刀、剑、斧，也不把自由锻造退化成预制刀身部件拼装。
- 不允许六维轻易全满，不把重量设计成单向越轻越好，不用万能总战力替代真实取舍。
- 不让故事编造成品数据里不存在的优缺点，不在运行时调用 LLM 代替因果规则。
- 不扩张到软体物理、全部冷兵器拓扑、任意焊接、多世界、经济成长、正式微信版或大量正式美术。

## 5. 固定架构

```text
玩家输入
  -> 输入适配器（键鼠，未来可换触屏）
  -> M1 锻造会话 / 纯 TypeScript 模拟（ForgeState）
  -> ForgeSnapshot（只读视觉快照）
  -> M6 Cocos 渲染

完成锻造
  -> M3 从最终状态推导 WeaponData
  -> NPC 规则代理读取 WeaponData + 世界状态 + 随机种子
  -> 可复现的冒险故事
```

公共结构：`ForgeState`、`WorkpieceState`、`BladeSection`、`JointState`、`ForgeOperation`、`ForgeSnapshot`、`WeaponData`。模拟层不能依赖 Cocos；参数和文案进入数据表。

## 6. 六周路线

1. S0：可迁移协作与工程地基。
2. S3a：最小物理内核和确定性回放。
3. S3b：双手操控、连续形变和即时反馈。
4. S3c：最小完整工序、隐藏六维、缺陷与回火。
5. S3d：标准护手/握柄受控装配，保留未来多材料接口。
6. S2a：一个规则代理世界、三类 NPC 委托和因果故事。
7. S6：完整闭环、试玩、公开 H5、录像、系统说明和迭代证据。

## 7. 协作方式

每个 2-4 天切片按 Issue → Goal → branch → Draft PR → 实现/测试 → 作者试玩 → Codex review → squash merge 进行。作者负责体验决策、Cocos GUI 首次操作和真人验收；Codex 负责代码、数据、测试、构建、Git 和问题定位。两台设备绝不同时开发，切换设备严格执行 `$device-handoff`。
