## 1. 项目定位与交付边界

**一句话定位**：玩家是一名铁匠，自由组合锻造工艺打造武器；武器由可解释的隐藏六维性能决定一段冒险故事，并可成为分享内容。

这是系统策划秋招作品集。优先级是“设计深度 + 一条完整、可玩的核心闭环”，不是商业化完整度。

两大卖点：

1. 自由、可重复、能看见因果的锻造过程：玩家敲在哪里、何时加热、如何淬火、如何研磨，钢坯就以对应方式变化。
2. 数值驱动、可解释的冒险故事：故事不是随机夸奖，而是武器状态的唯一明确反馈。

本轮最终产物必须能在**浏览器 H5**和**微信小游戏**运行；它们共享游戏逻辑、Three.js 场景、程序化模型、视觉参数和数据。浏览器是调试、自动验收与作品集入口，不是未来迁移前的临时原型。无需正式上线微信、支付、广告或服务端。

## 2. 玩家真实游玩流程

最终体验是固定斜俯视的 2.5D 铁匠铺。画面中没有玩家角色；玩家看见工作台、材料架、熔炉、铁砧、淬火槽和磨石等器具。

```text
进入工坊
-> 点击材料架：选择材料
-> 点击熔炉：加热并观察热色
-> 点击铁砧：钢坯置于铁砧面上，玩家令其转面并用锤子落锤塑形
-> 点击淬火槽：拖动钢坯完成淬火
-> 点击磨石：沿刃口研磨
-> 得到一把自由形状、带隐藏性能和事实缺陷的武器
-> 雇 NPC 冒险，观看武器造成的故事后果
-> 后期可分享给好友观看同一武器的故事
```

R1 不先制作完整工坊总览，而是用程序化近景依次跑通选料、加热、铁砧、淬火、研磨；R4 再把这些近景接回可点击的 2.5D 工坊中枢。

锻打近景的当前核心构图：铁砧居中、钢坯斜放于砧面、锤子在右上方；已确认移除钳子，钢坯须原地围绕自身长轴转面。PC 当前可鼠标短按轻击、长按蓄力、`A/D` 每次转面 90 度；W/S 送料尚未实现。微信后续使用拖动送料、明确转面控件、点击轻击、长按蓄力。两端输入都只能生成相同的 `ForgeIntent`，不能直接写游戏状态。

## 3. 设计原则：自由，但不黑箱

锻造是“操作为地基、工艺为配方”；故事是“武器数据为接口、内容为配方”。系统要让玩家组合出多样结果，而不是让玩家寻找唯一标准答案。

- **工作台状态机，不是固定流水线**：材料对象持有当前状态；工具只在状态合法时亮起。以后新增工艺，只增加操作和门槛，不推翻地基。
- **透明的自由**：自由不是瞎试。温度色、局部形变、裂纹、火花、蒸汽、研磨反光、声音等都要让玩家理解发生了什么；失败必须可读。
- **评判执行，不评判意图**：操作精度有客观后果，例如低温重锤更易积累损伤、淬火时机错误会影响热处理、研磨不均会留下刃口问题。玩家想锻成匕首、弯刀或奇怪的“像斧子的剑”都合法，系统不以预设造型评判其价值。
- **故事用后果说话**：玩家过程界面不显示“优秀/失败”总评和六维数字；最终由 NPC 冒险中的卷刃、断裂、失衡、威慑或劈砍后果揭示武器。

首版最小工艺链：

```text
选料 -> 加热 -> 锻打 -> 淬火 -> 研磨
     -> 隐藏六维与事实缺陷 -> NPC 规则冒险 -> 故事后果
```

回火、正火、受控装配、动力锤、图纸、折叠焊接和自由焊接都在最小链稳定后讨论；它们不得提前挤占核心验证时间。

## 4. 游戏物理、形变与六维评估

### 4.1 采用的“够用且可扩展”的模拟

不做有限元、软体物理或真实化学。我们模拟对玩家操作和故事后果真正有用的四类状态：

| 状态 | 保存什么 | 玩家能看见什么 |
| --- | --- | --- |
| 材料 | 密度、可硬化潜力、热参数、基础韧性等 | 材料外观、加热与成形手感差异 |
| 几何 | 截面宽度、厚度、长度分布、中心偏移、质量分布 | 刀尖、刀腹、刀背、刃口、弯曲、柄茎与重心 |
| 热处理 | 温度、塑性、淬火历史、后续回火预留 | 热色、冷却、蒸汽、硬化结果 |
| 完整性与表面 | 应力、损伤、裂纹、研磨连续性、表面完成度 | 裂纹、卷刃风险、磨痕、金属光泽 |

钢坯使用 **24 个物理截面**。每段记录左右宽度、上下厚度、中心偏移、局部体积、温度、塑性、应力和损伤；渲染时插值为约 96 个视觉截面，使刀坯连续平滑。首版支持刀尖、刀腹、刀背、刃口、弯曲和简单柄茎；不支持开孔、分叉和任意拓扑。

每次锤击只影响命中截面与邻近截面：热且可塑时更容易压薄、向周边延展；低温重击更容易积累应力与损伤；长期不对称受击会形成弯曲。近似体积守恒，不能凭空把钢敲没或敲多。所有操作必须确定性可回放。

这不是“认真热锻/乱敲”这样的主观按钮扣分。测试中可以用“正确操作序列、低温重击、单侧重击”等描述场景；游戏里结果只来自玩家实际输入和当前材料状态。

### 4.2 参数化伪形变与画面反馈

玩家需要相信“我砸这里，这里真的变了”，但不需要真实材料仿真。技术上用截面参数驱动 Three.js 动态网格：物理截面是事实，视觉截面只是平滑插值。

每次落锤必须在 100ms 内出现局部形变、热色变化、火花与锤击声；淬火出现蒸汽；裂纹有清晰的视觉与声音提示。连续 30 次锤击不得出现断层、空白或报错。若动态网格在两天内无法稳定，画面可降级为连接式 2D ribbon，但不得改变 `ForgeState` 与 `ForgeSnapshot` 接口。

研磨沿刃口滑动，先以程序化表面状态表现“暗哑覆盖被逐步磨开”：覆盖率影响刃口完成度，均匀度影响外观与局部缺陷。不是点击一下就直接加锋利。

### 4.3 隐藏六维是“翻译层”，不是根基属性

最终状态由 `evaluate` 模块推导为 `WeaponData`：

| 六维 | 主要来源 | 故事中的可见后果 |
| --- | --- | --- |
| 锋利 | 刃口几何、研磨连续性、硬度 | 切割、砍断、卷刃 |
| 硬度 | 材料与淬火状态 | 抗变形、硬碰硬、破甲 |
| 韧性 | 材料、热状态、损伤与裂纹 | 格挡、久战、恶劣环境、断裂 |
| 重量 | 密度与实际体积 | 跋涉、敏捷、偷袭、桥梁负担 |
| 平衡 | 质量分布与重心 | 连续动作、疲劳、失衡 |
| 外观 | 几何协调、表面完成度、工艺痕迹 | 威慑、社交、身份误判 |

六维对玩家隐藏，浏览器 `?debug=1` 仅供作者检查公式输入、结果和缺陷来源。`traits` 与 `flaws` 必须结构化保存，例如薄刃、前端过重、弯曲、裂纹、连接不良（后续装配阶段）。不设置“总战力”；若未来保留总评分，也只能用于展示门面，不能替代故事中的逐维判定。

物理取舍必须存在，不能把六维轻易全满：狠淬可提高硬度却增加脆性风险；更薄的刃口更利却更容易崩卷；减料可变轻却降低抗造能力。平衡和外观是“工艺把资源分配得是否协调”的结果，不是任意形状的惩罚。

特殊或魔法功能（如淬毒、引火、认主）不塞进物理六维，应通过工艺标签或后续故事规则单独表达。

## 5. 故事、成长与分享

### 5.1 最小故事闭环

R3 先做一个 NPC、三类情境、3 至 5 个确定性事件。故事只读取 `WeaponData` 和随机种子：同一武器与种子必须得到相同事件；故事不得读取玩家点击过程，也不得编造数据中不存在的缺陷。

故事的乐趣来自反差、因果、代入和稀有结果。例如外观出众但锋利差的武器可以威慑强盗，却砍不断障碍；自己造成的裂纹会在冒险格挡时断裂。冒险“难”主要是结局稀有度，而不是让大部分武器无法走完流程。

完整方向是轻跑团：`WeaponData + 世界状态 + 玩家选择 -> 条件规则 -> 事件 -> 结局`。不做剧情指数级大分叉；故事内容通过情境库、六维分段、条件文案和事件模板扩展。

### 5.2 成长与社交（后置）

后续材料与工具提升的是工艺上限和损耗控制，不应该硬锁基本操作。图纸是可忽略的新手提示，不是唯一合法配方。失败可以产出裂纹和滑稽故事，是内容而不是纯惩罚。

最终有两个出口：玩家雇 NPC 自己看故事，获得即时反馈；或分享给好友查看已确定但对好友未知的故事。R4 才处理微信分享数据、浏览器 URL/二维码与存档；本轮不做服务端防篡改。

## 6. 画面与美术方向

2.5D 是降低资产成本并保留真实感的方式，不是把游戏退回纯 2D。近景锻打强调金属形变、温度、火焰和工具；工坊总览强调器具可读性；冒险画面后期可用剪影分层强化武器焦点。

视觉借鉴《小丑牌》的优点：信息层次清楚、反馈鲜活、有独立气质、像手工打磨过；**不复制**其 2D 卡牌美术。目标是略带像素颗粒/后处理、清晰描边、金属高光、暖色炉火主光源与统一的锻造奇幻气质。避免纯低保真像素和纯扁平矢量。

首轮完全程序化生成，不使用外部模型或贴图。R4 再通过代码生成的描边、光照、色板、火花、蒸汽和程序音效建立统一风格。未来如引入正式美术，先做一个“风格样张”作为所有资产的参照；风格统一优先于单张精美。美术若无法稳定达到精致像素感，应退到“简洁形状 + 厚描边 + 暖色光影”，不为模仿某款游戏硬耗时间。

美术介入节奏：R1a 只保留可读的基础金属与热色，服务于操作验证；R1b 制作并由作者验收一张可玩的锻打近景构图样张，当前确认“铁砧居中、无钳子、钢坯原地转面、右上锤子”的空间关系，器具精细建模与统一风格仍后置；R4 才把已通过的方向扩展为完整工坊和故事呈现。平台头像不属于游戏内美术资产，不影响这条节奏。

## 7. 架构契约与数据归属

```text
BrowserRuntime / WechatRuntime -> GameApplication
平台输入 -> ForgeIntent -> ForgeOperation -> ForgeState
                                      |-> ForgeSnapshot -> render
                                      `-> evaluate -> WeaponData -> story
```

公共契约：

- `ForgeIntent`：平台无关的送料、转面、落锤、蓄力、拖动淬火、研磨等意图。
- `ForgeOperation`：合法性判断后的确定性操作记录，可保存和回放。
- `ForgeState`：材料、24 截面、热处理、完整性、表面、阶段和规则版本。
- `ForgeSnapshot`：仅供渲染读取的几何、热色、裂纹与表面数据。
- `WeaponData`：形状摘要、隐藏六维、`traits`、`flaws`、材料和规则版本；没有总战力。
- `StoryResult`：种子、实际事件、武器因果和结局。

材料参数、物理参数、六维映射、故事节点与文案必须进入带运行时校验的数据文件。纯逻辑模块禁止导入 Three.js、浏览器对象或微信 API。渲染只能读取快照，不能反向改数值。

## 8. 阶段计划与作者验收

### R0：双运行时技术门

目标：先证明 Three.js 方案在浏览器和微信小游戏都能使用同一份状态与渲染逻辑，不把玩法建立在未知兼容性上。

交付：固定斜俯视相机、程序化灯光、铁砧、24 截面钢坯；浏览器点击或微信触摸钢坯后局部截面确定性变化。

验收：浏览器自动测试通过；微信开发者工具可编译显示；作者真机预览可触摸并看到变化；两端回放同一操作得到相同 JSON；微信包体合规；Windows 全新 clone 能重新构建两个产物。

### R1：最小可玩锻造链

目标：跑通选料 -> 加热 -> 锻打 -> 淬火 -> 研磨，先不做工坊总览、故事、回火、正火、装配或正式美术。

交付：两套输入产生同一种意图；局部锤击影响相邻截面；温度、形变、应力、损伤、淬火和研磨状态可见；每次锤击 100ms 内有反馈；同一回放跨平台一致。

验收：作者可连续锻打 30 次不出错；高温与低温、对称与单侧锻打产生肉眼可见且数据可解释的不同；完整最小链能导出一份可评估的最终状态。

### R2：物理规则与隐藏六维

目标：把 R1 的最终状态稳定地翻译成六维、特性和缺陷，证明没有简单“全满神刀”。

交付：数据校验、固定样本、`?debug=1` 作者调试页与自动测试；每个缺陷都有可追踪的状态来源。

验收：同一操作序列恒定；改变物理规则后标准样本能解释变化；极端/怪异外形仍合法；六维取舍与缺陷符合预期；玩家生产界面不显示数字评价。

### R3：最小故事闭环

目标：一把真实锻出的武器能进入一段因果一致的 NPC 冒险。

交付：一个 NPC、三类情境、3 至 5 个确定性事件与结局；故事只读 `WeaponData`。

验收：至少三套不同操作得到明显不同且可解释的故事；薄刃、厚钝、前端过重、弯曲、裂纹等会以合理后果进入故事；同一武器和种子结果一致。

### R4：正式体验与作品集

目标：把已验证的链条做成完整可展示体验。

交付：2.5D 工坊总览、器具入口、反馈打磨、程序化视觉风格、存档与分享数据、试玩记录、演示录像、规则样本与系统设计复盘。

验收：至少 5 名陌生玩家试玩；4/5 能从故事指出一个自己锻造造成的后果；3/5 第二次锻造会主动调整策略；怪异武器可进入故事而不是报错或被强行纠正。

## 9. 微信适配止损线

微信与浏览器有不同运行入口，但共享 `GameApplication`、`forge`、`evaluate`、`story` 和 `render`。渲染器显式接收画布与 WebGL 上下文，游戏自行驱动帧循环。
