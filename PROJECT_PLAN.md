# 《打了个铁》项目计划

> 唯一现行计划。它只保留当前决定、未决关卡和可执行验收；被删除的旧方案可从 Git 历史追溯，不能作为实现依据。

## 1. 目标与完成定义

六周内交付一个可在浏览器运行的 PC 键鼠 H5 Demo，用于展示系统策划能力。闭环必须成立：

```text
自由锻造连续坯料 -> 隐藏性能与缺陷 -> NPC 自主冒险 -> 可解释的故事后果
```

最终交付：可公开试玩的 H5、至少五名陌生玩家的记录、演示录像、Release、架构图、规则表、标准样本和迭代案例。玩家应能在故事中指出至少一个自己锻造造成的后果；至少三名玩家第二次会主动改变锻造策略。

首版不做：软体/有限元物理、完整冷兵器拓扑、任意焊接、运行时 LLM、多世界、经济成长、正式微信版、自由镜头、完整角色绑定、可漫游地图和大量正式美术。

## 2. 已锁定的产品决定

- 玩家像《锻刀大会》一样操作：W/S 送料，A/D 每次旋转 90 度，鼠标点击轻击、长按蓄力。
- 最小完整链：**选料 -> 加热 -> 锻打 -> 淬火 -> 研磨**。它稳定后才加入回火、正火和受控装配。
- 锻造是确定性规则模拟，不是按钮加属性，也不是工业级物理。首版模拟温度、塑性、截面几何、近似体积、应力、完整性、弯曲、裂纹及后续热处理状态。
- 自由连续坯料合法且可进入故事；没有 `baseForm`、总战力或强制分类。标准护手和握柄只在 S3d 加入。
- 六维（锋利、硬度、韧性、重量、平衡、外观）和 `craftsmanship` 从最终状态推导。六维不在锻造界面展示；`craftsmanship` 不是战力总分。
- 缺陷由状态阈值确定性产生。故事由规则代理的实际事件生成，同一武器、世界初态和种子必须重放同一结果。
- Demo 是桌面浏览器 H5；未来可迁移微信，但本轮不做微信工具链或经济/社交服务。

## 3. 玩家体验与 2.5D 方向

本作是原创 2.5D 锻造工坊：Cocos 3D 场景承载工作区、空间、火光和工具；刀坯是截面状态生成的轻量 3D 外观；UI 是 2D 叠层。它不是纯 2D 卡牌游戏，也不是自由镜头的全 3D 模拟器。

《小丑牌》仅是独立游戏设计参考：信息层级清楚、UI 有材质和性格、反馈强、低成本资产仍统一。不得把它当作最终风格标签或模仿其具体画面、卡牌、字体、图案和提示词。

每次锻造操作必须有玩家可读的物理反馈：热色和亮度提示可塑性；形变、火星、声音与震动感提示锤击；裂纹、弯曲、蒸汽与表面变化提示风险。不得用“硬度 +2”“成功/失败总评”取代这些反馈。

### 2.5D 技术边界

- 一个固定工作区；静态环境优先，动态效果受 H5 性能预算限制。
- 8-10 个截面优先生成有正面、背面和侧面的刀坯网格。两天内不稳定时，用同一快照驱动置于 3D 场景中的 ribbon，不改变模拟接口或整体 2.5D 体验。
- 工坊镜头建立空间感；工序近景必须看清宽、厚、弯曲、热色和裂纹。
- S3b 先用灰盒工作区、程序刀坯和基础特效验证手感；正式资产只在样张通过后生产。

### 未决设计关卡

以下事项尚未讨论完成，不能由实现者擅自决定；它们必须在对应切片前由作者确认并写回本文件：

1. 工坊的时代、地域、世界观与总体情绪。
2. 玩家只见工具、局部手臂或完整铁匠角色。
3. 主镜头与各工序近景的机位、切换与运动。
4. 最终模型密度、纹理笔触、描边、色板和后处理。
5. 故事使用剪影事件卡、连续分镜或其他呈现方式。
6. 锤击、火炉、淬火、研磨的声音方向和来源。
7. 首次玩家的引导、重开/交付流程与单局时长。

正式美术前需要通过一张样张：炉火、铁砧、红热刀坯和一块 UI 同画面，证明焦点、金属/高温可读性、2.5D 空间与 UI 协调。采用资产必须记录来源、授权、尺寸、用途和样张版本；动态刀坯及状态效果始终由 Cocos 生成。

## 4. 系统架构

```text
键鼠 -> M10 input -> M1 forging -> ForgeOperation -> ForgeState
                                                -> ForgeSnapshot -> M6 render

完成锻造 -> M3 evaluation -> WeaponData -> M4 agent world -> M5 story
```

| 模块 | 责任 | 禁止 |
|---|---|---|
| M1 forging | 会话、合法操作、纯模拟推进 | Cocos、六维、故事 |
| M3 evaluation | 从最终施工状态推导六维、完成度、特征/缺陷 | 改 ForgeState、按钮加属性 |
| M4 agent world | NPC、世界状态、确定性事件 | 读锤击过程、编造武器事实 |
| M5 story | 将已发生事件变为条件文案 | 决定模拟结果 |
| M6 render | 读取快照渲染 2.5D 形变和反馈 | 反写状态 |
| M7 scene | 场景与流程导航 | 物理或故事规则 |
| M8 save/share | 存档、重放、分享摘要 | 改变结果 |
| M9 data | 参数、材料、委托、文案 | 隐藏运行逻辑 |
| M10 ui/input | 输入适配、显示 | 业务判定 |

公共数据：`ForgeState`、`WorkpieceState`、`BladeSection`、`JointState`、`ForgeOperation`、`ForgeSnapshot`、`WeaponData`。纯模拟、评估和代理世界只能依赖 TypeScript/数据文件，不能 import Cocos。

### S3a 最小契约

`ForgeState` 可序列化并保存参数版本、初始材料、工件、工序阶段和不可变操作记录。`BladeSection` 至少有位置、段长、宽、厚、局部温度、塑性、应力、完整性和横向偏移。S3a 只实现 `heat`、`rotate`、`hammer`；`quench`、`grind` 只作为未来操作类型预留。它不做 Cocos、六维、淬火规则、研磨规则、回火、正火或装配。

S3a 的自动样本必须覆盖：材料可锻温区内高温更易形变、低温更易积累应力、重新加热可释应力但不能修复损伤、过热累积热损伤、90 度转面交换宽厚响应、邻段平滑、受击后的横向扩展与沿坯料长度延展、体积误差阈值、单侧受击弯曲、由低塑性下的局部重复形变、截面变薄与既有热损伤共同决定的确定性裂纹，以及 JSON 回放。

### 遗留代码迁移

现有 `assets/scripts/weapon` 与 `assets/data` 是旧 S1 骨架，不是现行合同。S3a 建立新契约测试后，再替换 `baseForm`、`overall`、直接材料基础六维和固定故事判定；旧文件在其引用被迁走后删除或改为明确的只读样本。新功能不得依赖它们。

## 5. 执行节奏与验收

### S0a：计划与协作基线（PR #2）

把仓库真相收敛为 `AGENTS.md`、`CURRENT_TASK.md`、本计划和两份专项文档；建立 Issue/PR、`main` 保护、设备交接 Skill 与大文件规则。S0a 只交付可审查的项目基线，不假装 Cocos 工程已完成，也不自动关闭完整 S0 的 Issue #1。

通过：作者确认计划和分支/PR/main 的关系；Codex review 无阻断项；PR #2 squash merge 后 GitHub 默认 `main` 展示现行文档。

### S0b：可迁移 Cocos 工程（macOS 剩余约三天）

导入 Creator 3.8.8 3D 工程骨架；提交 `assets/`、全部 `.meta`、`settings/`、`package.json` 和锁文件，忽略生成目录。建立 Node 开发依赖、类型检查、单元测试、数据校验、`doctor`、Web Desktop 构建包装和只跑纯逻辑的 GitHub Actions。构建第一份 H5，安装 LFS，并从全新目录 clone、安装、检查、打开 Creator、构建 H5。CI 首次通过后设为 `main` 必需检查。

通过：新目录不依赖旧缓存；作者可打开工程、预览/构建 H5；生成目录未进 Git；远端、分支和工作区清楚。

### S3a：最小物理内核（2-4 天）

实现上述纯 TypeScript 契约、回放和标准样本。作者验收数据趋势，不要求 Cocos 场景。

通过：相同序列深度相等；认真、乱敲、冷敲、单侧猛敲有不同且可解释的结果。

### 设计关卡 A：S3b 前

先确认“未决设计关卡”中会阻塞灰盒交互的工坊基调、角色露出、镜头与首次引导。只确认首轮需要的答案，不提前生产最终资产。

### S3b：2.5D 锻打交互（2-4 天）

在灰盒工作区接入 W/S、A/D、轻击和蓄力；在接入输入前为纯 M1 增加确定性的时间推进/被动冷却接口，使温度随工序时间下降；快照生成连续刀坯并在 100ms 内给出温度色、火星、声音、蒸汽或裂纹反馈。验证镜头和动态网格，必要时在两天内降级 ribbon。

通过：连续 30 击无空白、断层或报错；四面宽厚正确；工坊不遮挡落点；H5 桌面目标 60 FPS。

### S3c：最小完整链与隐藏评估（4-6 天）

加入选料、淬火、研磨；M3 从最终状态推导隐藏六维、`craftsmanship`、结构化 `traits/flaws`。最小链稳定后再加回火，正火留在回火之后评估。

通过：六维不能轻易全满；重量和外形有不同用途；规则修改能由固定样本解释；交付前不显示数值结算。

### S3d：受控装配（2-3 天）

给自由刃体接入标准护手与握柄；连接位置、接触面积、质量与重心进入状态。接口保留多个 `WorkpieceState` 和 `JointState`，但没有任意焊接 UI。

通过：装配不覆盖刃体；连接差可成为后续故事事实；未来焊接不必重写 M3/M4。

### 设计关卡 B：S2a 前

用真实锻造样本决定三类 NPC 委托、武器事实到事件的映射、故事视觉语言和首版文案体量。不得先写大量脱离真实武器分布的剧情。

### S2a：规则代理世界（5-7 天）

一个世界、三类 NPC 委托。NPC 有目标、性格、体力、伤势、位置和关系；它们根据自身/世界/武器状态行动，M5 只讲已发生的事件。

通过：薄刃、厚钝、前重、弯曲、裂纹和连接差都有实际后果；怪异成品不非法；同一输入和种子完全重放；不存在的缺陷不能被文案说出。

### S6：整合与作品集交付（最后一周）

打通锻造、交付、冒险和分享重放；完成陌生试玩、阻断修复、公开链接、Release、录像、系统架构、规则表、数值样本和迭代复盘。

通过：至少 4/5 玩家能从故事指出锻造后果，3/5 会改变第二把的策略，怪异成品不会报错或被矫正。

## 6. 工程、Git 与跨设备

- 每个 2-4 天切片：Issue -> Goal -> branch -> Draft PR -> 测试/试玩 -> review -> squash merge。一个 PR 可跨模块但只交付一条玩家行为。
- 每个公共接口先有契约测试；参数、文案和材料进入数据文件。CI 只跑 TypeScript/数据，Cocos/H5 由本地实际验证。
- Cocos 项目只提交源文件和设置；大源资产走 LFS，构建包/视频走 Release。
- Cocos 自动化默认使用可审查的项目文件、官方命令行构建和 H5 浏览器验收。批量场景编辑成为瓶颈时，优先编写仓库内的最小 Creator 扩展；社区 MCP 只能先做隔离评估，不能成为工程必需依赖。
- 账号登录、系统授权和真人试玩由作者操作。Codex 负责代码、数据、场景脚本、构建、日志和可重复验证，不要求获取账号或高权限桌面控制。
- 两台设备永不同时开发。离开前关闭 Creator、检查、更新 `CURRENT_TASK.md`、commit、push、核对远端、清理工作区。Windows 第一天只复现环境和 H5，不开发新功能。

## 7. 计划审计结论

计划可以执行，前提是按本文件的关卡推进，而不是把历史愿景一起实现。已消除或隔离的风险：

- 经济、图纸成长、预制刀身和显示六维从首版范围移除。
- S3a 只做热/锤打内核，热处理与评估在 S3c，避免一个切片承担全部物理。
- 2.5D 工作区是灰盒交互的技术基础；最终美术必须经样张门槛，不阻塞 S0/S3a。
- 遗留 S1 数据被明确隔离，避免旧 `overall`/`baseForm` 污染新合同。
- NPC 内容设计被放到真实武器分布之后，避免脱离数据空写故事。
- 六周安排保留最后一周整合和试玩；任何未通过的样张或技术验证按降级方案收敛，不扩展范围。
