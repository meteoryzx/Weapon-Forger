# 《打了个铁》AI 工作规范

> **本文档定位**：编码 AI（Codex 等）的工作规范。项目玩法/美术/执行计划见另一份《打了个铁·项目全案》（作者用）。
> 作者不碰代码，是"AI 的架构师 + 质检员"。本文所有内容为约束 AI 稳定产出、且不看代码即可验收而设计。
> **PROJECT_RULES.md / DECISION_LOG.md 机制尤其重要**——是新 agent 冷启动理解项目、不违反架构的唯一依据。
> 平台：微信小游戏 | 引擎：Cocos Creator | 语言：TypeScript

## 0. 本项目的开发范式：Prompt 工程（先读这段，它决定一切）

**你的真实角色**：不是程序员，是**AI 的架构师 + 质检员**。你的工作 = 用文档约束 AI 稳定产出 + 验收对错。

**由此推出的最高铁律（所有规则的根）**：
1. **一切可验证化**：你不看代码，AI 每交付一块，必须给"你能看懂的证据"（可点的 demo / 录屏 / 数据日志），而非"我写完了"。
2. **模块可独立理解/修改/验证**：拆到 AI 改 A 模块时无需读 B 模块——这是给"不碰代码的你"买的保险，出错只坏一小块。
3. **接口先行**：先定死模块间怎么对话，再让 AI 填内部；内部写乱了，重写单模块即可，不牵连全局。
4. **配置与逻辑分离**：材料/判定节点/故事文案做成数据表，**你自己改，不求 AI**。
5. **小步快跑**：每次只让 AI 做一个可独立验证的小块，验过再下一个。绝不"一次一大坨"。
6. **AI 有记忆锚点**：项目规约 + 决策日志，AI 每次开工先读，防止自由发挥违反架构。

> ⚠️ **你必须接受的现实**：不碰代码 = 排错能力受限。缓解全靠"模块化 + 小步验证"——出错时是"某小块坏了"，而非"整个项目跑不起来还不知哪错"。

## 1. 模块划分与职责边界

> 原则：高内聚、低耦合**是为了 AI**（上下文有限），不是为优雅。每个模块"一句话职责 + 明确不做什么"。

| 模块 | 一句话职责 | 明确不做什么 |
|---|---|---|
| **M1 锻造交互** | 处理点/拖/滑手势，把玩家操作转成对材料状态的修改 | 不算最终数值、不碰故事、不画渲染 |
| **M2 武器数据（WeaponData）** | 唯一真相源：存武器的形态参数+6维+材质+工艺标签+裂纹等状态 | 不含任何逻辑，纯数据结构 |
| **M3 数值评估** | 把材料状态与操作记录，算出 6 维等级 + 总评分。**Demo 期用"简化取舍公式"（三对物理对立，见全案§3.1b）；未来可整体升级为"物理模拟"（4根基量涌现），但对外仍只输出 6 维——升级只改本模块内部，接口不变** | 不改外形、不管显示 |
| **M4 冒险判定** | 读 WeaponData 的 6 维，按"判定节点表"跑出冒险分支结果 | 不关心武器怎么锻的、不管文案排版 |
| **M5 故事呈现** | 把 M4 的判定结果 + 文案表，渲染成 PPT 式冒险故事 | 不做判定逻辑，只负责"讲" |
| **M6 渲染表现** | 根据 WeaponData 画出武器/刀坯的 2.5D 外观、光效 | **禁止反向写任何数值/状态**（只读） |
| **M7 场景导航** | 2.5D 铁匠铺中枢，工具入口，进出各工序小游戏 | 不含工序内部逻辑 |
| **M8 存档/分享** | 存读进度；生成/解析二维码分享（传 6 维参数） | 不含玩法逻辑 |
| **M9 数据配置** | 材料表/工具表/判定节点表/故事文案表（**你直接改**） | 纯数据，无代码 |
| **M10 UI** | 数值条、对话框、按钮等界面 | 不含业务逻辑 |

## 2. 依赖关系与数据流向（禁止的耦合写死）

```
玩家手势 → [M1锻造交互] → 写 → [M2 WeaponData] ← 读 → [M3数值评估] 写回6维
                                      │(只读)
                    ┌─────────────────┼──────────────────┐
              [M6渲染](只读)     [M4冒险判定](只读)    [M8存档分享](只读)
                                      │ 读 M9判定节点表
                                      ↓
                                [M5故事呈现] 读 M9文案表
```

**铁律（AI 绝不能违反，写进规约）**：
- **数据只从中枢（M2）单向流出给"消费方"**（渲染/判定/存档只读 M2）；
- **只有 M1 和 M3 能写 M2**，其他模块一律只读；
- **M6 渲染层绝对禁止反向写数值**（避免"画面改数值"的混乱）；
- **M4/M5 不准直接读 M1**（判定只认最终数值，不认"怎么锻的"）；
- 跨模块通信走**事件总线**（如 `EventBus.emit('weapon_forged', weaponData)`），不直接互相调用内部函数。

## 3. 接口契约（先定死，再让 AI 填内部）

**核心数据结构 WeaponData（M2，全项目的"普通话"）**：
```
WeaponData = {
  shape: { 控制点[], 各段厚度[] },      // 外形（M1写，M6读）
  material: 材质ID,                     // 选材（M1写）
  process: [工艺标签...],               // 折叠/覆土/淬火介质（M1写）
  stats: { 锋利,硬度,韧性,重量,外观,平衡 }, // 6维等级0-10（M3写，M4/M6读）
  overall: 总评分,                      // M3写，M6/M8读
  flaws: [裂纹...],                     // 负面状态（M1写，M4/M5读）
}
```
> 这个结构**一旦锁定，就是所有模块的契约**。AI 改任一模块，只要不破坏这个结构，就不会连锁崩。修改它须走"决策日志"记录并通知全模块。

**关键事件（示例，实现时细化）**：`operation_applied` / `weapon_forged` / `adventure_resolved`。

## 4. 配置与逻辑分离清单（你的掌控区）

以下**全做成独立数据表（JSON/CSV），你自己维护，AI 只写"读表逻辑"**：

| 数据表 | 内容 | 谁维护 |
|---|---|---|
| 材料表 | 每种材料的名称/等级/对6维的基础影响/特性文案 | **你** |
| 工具表 | 每种工具/等级 → 提升哪个工序的上限与成功率 | **你** |
| 判定节点表 | 每个冒险节点：判定哪个维度/分段区间/各分支指向 | **你** |
| 故事文案表 | 每个节点×分段 → 对应叙事文字；各结局文案 | **你** |
| 图纸表 | 工艺流程提示（大马士革等） | **你** |

> 意义：**你的策划能力直接作用在数据上，不经过代码、不求 AI。** 内容迭代零摩擦。

## 5. 面向 AI 的目录结构（让 AI 看懂全貌、安全改任一模块）

```
project/
├── PROJECT_RULES.md        ← AI 每次开工必读（架构铁律+接口+目录说明）

├── assets/
│   ├── scripts/
│   │   ├── forging/        ← M1 锻造交互
│   │   ├── weapon/         ← M2 WeaponData（纯数据）
│   │   ├── evaluation/     ← M3 数值评估
│   │   ├── adventure/      ← M4 冒险判定
│   │   ├── story/          ← M5 故事呈现
│   │   ├── render/         ← M6 渲染
│   │   ├── scene/          ← M7 场景导航
```

## 6. AI 协作工作流（怎么派活、怎么验收）

**每个模块的标准流程**：
```
① 你派活：指明做哪个模块 + 要求先读 PROJECT_RULES.md
② AI 先复述：它对该模块职责/接口的理解（防跑偏，你确认）
③ AI 实现：只动这一个模块
④ AI 交付"可验收证据"：一个能点的最小 demo / 录屏 / 数据打印
⑤ 你验收：对照本模块"验收标准"（见 §7），只看行为不看代码
⑥ 过 → 记决策日志，下一个模块；不过 → 指出现象，AI 修
```

**派活 Prompt 模板**：
> "现在实现 [M4 冒险判定]。先读 PROJECT_RULES.md 和 data/判定节点表，复述你对该模块职责和接口的理解，我确认后再动手。只改 adventure/ 目录。完成后给我一个能输入6维数值、输出冒险分支结果的可点测试页，我要自己点着验证。验收通过后按 §6.1 提交 git。"

### 6.1 Git / GitHub 习惯（AI 必须遵守，写进 PROJECT_RULES）
> 目的：给"不碰代码、排错弱"的你**买后悔药**——任何时候都能回退到上一个能跑的版本。这是你这种开发模式最重要的安全网。

- **仓库**：项目第一步就建 git 仓库 + 连 GitHub 私有远程库（见执行计划第 0 步）。
- **提交时机 = 每个模块/小步"验收通过"后立即提交一次**（一个能跑的检查点 = 一次 commit）。**绝不攒着一大堆改动才提交**——攒得越多，出错越难回退。
- **提交信息格式**：`[模块] 动作`，例：`[M4] 冒险判定跑通，验收通过`。让你翻历史时一眼看懂每个版本是什么。
- **每次验收通过后推到 GitHub**（`git push`），异地备份，换电脑/换 agent 都能接着干。
- **AI 每次开工前先 `git status` 看有没有未提交的脏改动**，避免在别人没提交的半成品上乱改。
- **出问题的救命操作（你要会这一条）**：如果 AI 把某模块改坏了、又修不回来，你直接让它"回退到上一个提交"（`git reset --hard`到上个能跑的 commit）——这就是为什么每步都要提交。

### 6.2 其他开发好习惯（AI 必须遵守）
1. **一次一个模块，绝不并行开多个**：符合"小步快跑"，也让每次 git 提交对应一件清晰的事。
2. **改动前先说计划、你点头再动手**（§6 流程的②）：防止 AI 自作主张大改。
3. **不删你看不懂的东西**：AI 要删文件/大段代码前必须先问你。
4. **所有"数字/文案/规则"进数据表，不写死在代码里**（§4）：保证你能自己改。
5. **每个模块留一个"测试入口"**：一个能独立点、独立验的小页面/按钮，别依赖整个游戏跑起来才能测（这是你验收的抓手）。
6. **报错要"翻译成人话"**：AI 遇到问题，要用你能懂的话说"哪里、为什么、我打算怎么办"，不甩一堆报错日志。
7. **依赖/版本锁定**：引擎版本、第三方库版本写进 PROJECT_RULES，换环境不跑偏。
8. **README 常新**：每完成一步，更新"当前进度 + 下一步"，方便任何 agent 冷启动接手。

## 7. 各模块验收标准（你不看代码，靠这个判断对错）

| 模块 | 你怎么验（不看代码） |
|---|---|
| M1 锻造交互 | 能点/拖/滑，操作后材料状态数值实时变化（屏幕打印出来看） |
| M3 数值评估 | 手动造几组极端武器，看6维算得合不合理 |
| M4 冒险判定 | 输入一把武器→输出的分支是否符合判定节点表 |
| M5 故事呈现 | 同一把武器每次生成的故事一致；不同武器故事不同 |
| M6 渲染 | 改WeaponData参数→画面跟着变；确认它没偷偷改数值 |
| 整合 | 走通"选材→锻打→打磨→出武器→雇NPC冒险→看故事"完整闭环 |

## 8. 施工顺序（小步快跑，每步可验证）

> 原则：**先做能独立验证核心命题的模块，美术和内容最后填**（Balatro 经验：核心机制先定死）。
> ⚠️ 本表是技术侧对应；给作者看的大白话路线图见《打了个铁·项目全案》第四部分。

| 步 | 做什么 | 验证的核心命题 | 前置依赖 |
|---|---|---|---|
| **S0** | 项目初始化：git+GitHub、Cocos 空工程、PROJECT_RULES.md、DECISION_LOG.md、目录骨架、锁定 WeaponData 契约 | 环境能跑起来、规约就位 | 无 |
| S1 | M2 WeaponData + M9 数据表骨架 + **你手填 3–5 条测试数据**（几种材料、2–3 个判定节点、几段文案） | 数据结构立住、有料可测 | S0 |
| S2 | **M4 冒险判定 + M5 故事呈现**（纯逻辑，无美术） | **卖点二：数值真能驱动出不同故事**（最先验证，成本最低） | S1（依赖测试数据） |
| S3 | M1 锻造交互 + M3 数值评估（先粗糙，不调手感） | 锻造操作真能产出不同数值的武器 | S1 |
| S4 | M6 渲染（武器可视化）+ M7 场景导航 | 画面串起来 | S3 |
| S5 | 调锻打/打磨**手感**（此时机制已通，专心磨手感）🔴难点 | 卖点一：好玩 | S4 |
| S6 | M8 存档 + 二维码分享 + M10 UI 打磨 | 社交闭环 | S2+S5 |
| S7 | 填内容（材料/节点/文案表）+ 美术资产接入 | 丰富度 | S6 |

> **两个关键点**：
> - **S0 是新增的前置**——没有它，S1 无处安放。
> - **S1 必须你手填一点测试数据**，否则 S2 的"数值→故事"跑不起来（这是之前的逻辑断层，已补）。
> - **S2 先行**：用纯逻辑最快验证你最不确定的"数值→故事"，不成立就早调，避免美术白做。

> 🔴 **全局唯一真难点标注**：**S5 锻打/打磨手感调校**——技术方案已定（参数化伪形变，见《项目全案》§2.5），但"好不好玩"只能反复试玩磨，预留充足迭代时间，别指望一次到位。

## 9. AI 记忆锚点机制

- **PROJECT_RULES.md**（AI 每次开工必读）：§1 模块职责 + §2 数据流铁律 + §3 接口契约 + §5 目录规范 + §6.1/6.2 git 与开发习惯。开工 Prompt 第一句永远是"先读 PROJECT_RULES.md"。
- **DECISION_LOG.md**：每定一个规矩/改一次接口就追加一条。AI 遇到不确定先查它。
- 二者就是塔防/策划团队两篇文章里"为 AI 设计的记忆"，是你这种模式防止 AI 失忆乱写的核心。

- 核心机制最早原型就定死，内容打磨在后 → 对应本文 §8 施工顺序。
