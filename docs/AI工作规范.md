# 《打了个铁》AI 工作规范

> 最后更新：2026-07-20。本文件解释 AI 怎样稳定交付；架构硬规则以根目录 `PROJECT_RULES.md` 为准，当前任务以 `CURRENT_TASK.md` 为准。

## 1. 作者与 AI 怎样配合

作者不写代码，但不是旁观者。作者负责定义玩家体验、决定取舍、操作 Cocos GUI、亲手试玩并批准结果。AI 负责实现、数据、测试、构建、Git、排错和 review。

AI 每次交付都要回答四件事：

1. 玩家现在能做什么或看到什么？
2. 哪些证据说明它不是假装完成？
3. 这一步没有做什么，为什么？
4. 作者怎样在不读代码的情况下验收？

## 2. 一轮任务流程

1. 从 Issue 读取目标、边界和验收标准。
2. 设置当前 Goal，创建短期 branch；禁止在 `main` 开发。
3. 第一个检查点 push 后建立 Draft PR。
4. 先写或更新公共契约测试，再实现纯逻辑，再接 Cocos。
5. 每个可描述检查点 commit + push；未验收用 `[WIP]`。
6. CI 检查纯 TypeScript 和数据；本地检查 Cocos/H5。
7. 给作者明确入口和预期现象，作者亲手试玩。
8. AI 按 code review 方式检查整个 diff，优先报告 bug、回归、边界和缺测。
9. 全部通过后将 Draft PR 转正式，squash merge；给大里程碑打 tag。

AI 每次只解释当下 1-2 个新术语。命令由 AI 执行，作者理解目的和风险即可。

## 3. 纵向切片，而不是按模块排队

开发单位是一条玩家可验证行为，例如“同样一锤，高温坯更易变形且低温坯积累更多应力”。为了交付它，PR 可以同时改模拟接口、材料参数和测试。

禁止把 M1、M3、M6 分别放在长期分支开发，再到最后拼装。那会让公共接口漂移、冲突集中爆发，并且几周都没有可玩的东西。

合格 Issue 必须写：

- Player Outcome：玩家结果。
- In Scope / Out Of Scope：做与不做。
- Interfaces Touched：涉及的公共结构。
- Automated Acceptance：机器能检查的事实。
- Cocos / H5 Acceptance：作者能操作的事实。
- Risks And Recovery：最大风险与安全回退点。

## 4. 工程架构

### 4.1 施工期

```text
输入适配器 -> ForgeOperation -> 锻造模拟 -> ForgeState
                                      -> ForgeSnapshot -> Cocos 渲染
```

- `ForgeOperation` 是一条不可变操作：加热、锻击、旋转、淬火、研磨等。
- `ForgeState` 保存完整施工状态和参数版本，是回放与测试真相。
- `WorkpieceState` 表示一块材料；首版通常一块，接口允许未来多块。
- `BladeSection` 是连续坯料的局部截面，保存位置、宽、厚、局部温度/应力/完整性等。
- `JointState` 表示装配或未来焊接连接。
- `ForgeSnapshot` 是只读视觉投影，可为性能压缩，不能被渲染修改。

### 4.2 成品期

```text
最终 ForgeState -> M3 evaluation -> WeaponData -> 规则代理世界 -> 故事事件 -> 条件文案
```

`WeaponData` 至少包含：版本、材料摘要、几何摘要、质量/重心、六维、`craftsmanship`、结构化 `traits/flaws` 和连接摘要。它不依赖预设武器类别，也不携带 Cocos 对象。

### 4.3 依赖方向

- 纯模拟、评估、数据校验和世界代理只依赖 TypeScript/JSON。
- Cocos 层可以依赖纯逻辑；纯逻辑不能反过来依赖 Cocos。
- 故事只能读最终 `WeaponData`，不能用“玩家敲了很多次”直接写成性能好。
- 跨模块优先传数据和显式函数，不为了“解耦”过早引入全局事件总线或服务定位器。

## 5. 最小物理内核怎样把控

这里的“模拟”是可解释规则，不是连续介质求解器。每条规则必须回答：

| 问题 | 例子 |
|---|---|
| 现实依据是什么？ | 高温提高塑性，低温重击更容易积累损伤 |
| 玩家怎样影响它？ | 选择加热时机、受击段、方向和力度 |
| 画面/声音怎样回应？ | 形变量、颜色、火花、声音发生对应变化 |
| 数据怎样验证？ | 相同操作下比较高温与低温样本 |
| 参数怎样调？ | 阈值和曲线在版本化数据表中 |

首版规则：温度、塑性、四面受击、局部截面、近似体积守恒、应力、弯曲、确定性裂纹、淬火、回火、研磨和连接。先用 8-10 个截面；先保证趋势和因果正确，不伪装工业级精度。

裂纹等缺陷必须确定性产生。任何随机冒险内容都显式使用种子，保证 bug 能重放。

## 6. 视觉反馈与 Cocos

- W/S 沿刀坯送料；A/D 每次旋转 90°；鼠标点击轻击、长按蓄力。
- 锤击改变当前截面和邻近截面；网格从状态重新生成，不把节点变换当物理数据。
- 击打后 100ms 内出现视觉或声音回应。
- 四面旋转后，宽度与厚度投影要正确；连续 30 击无断层、空白或报错。
- Web Desktop 目标 60 FPS。动态网格两天仍不稳定就用连接式 2D ribbon，同一快照接口保持不变。
- 锻造 UI 不显示六维、裸物理数值和“优秀/失败”；可用火色、形变、声音、蒸汽和裂纹等世界内反馈。

## 7. 数值与故事质量

M3 只从最终状态推导：锋利、硬度、韧性、重量、平衡、外观。`craftsmanship` 是工艺完成度，不是战力汇总。`traits/flaws` 使用结构化 ID + 强度/位置/证据，文案不能只靠自由字符串猜测。

平衡检查至少包含：

- 标准认真样本、乱敲样本、低温重击样本、单侧猛敲样本。
- 六维不能轻易全满；重量有用途区间；不同几何适合不同情境。
- 修改一条规则时，固定样本对比能指出变化来自哪条规则。

世界代理必须先产生规则事件，再由 M5 选择文案。薄刃、厚钝、前重、弯曲、裂纹和连接不良都要有实际行为后果；“像斧子的剑”不非法，只按真实性能产生合理或滑稽结果。同一武器、世界初态和种子必须复现。

## 8. 自动检查分层

- 单元测试：单条物理规则和评估边界。
- 回放测试：完整操作序列的确定性结果。
- 数据校验：材料、参数、委托和文案的 schema/引用完整性。
- 契约测试：`ForgeSnapshot`、`WeaponData` 和故事事件边界。
- Cocos 验收：输入、连续形变、帧率、报错和 H5 构建，由本地执行。
- CI 只跑不依赖 Cocos 编辑器的检查，避免假装云端能验证引擎行为。

失败的检查必须原样记录为阻塞证据，禁止写“应该没问题”。

## 9. Git 安全与换机

- 离开设备前执行 `$device-handoff`；新设备第一天只复现，不开发新功能。
- 每次先检查当前 branch 和文件变化；按目的暂存，不盲目 `git add .`。
- 未完成但需要换机时，做 `[WIP]` commit 并 push，不把半成品合入 `main`。
- 日常禁止 `git reset --hard`、force push 和删除远端分支。先建 rescue 分支保存现场；正式历史回退用 `git revert`。
- Cocos 项目提交源文件、`.meta`、设置和依赖锁；忽略生成目录。大源资产 LFS，构建与录像 Release。

## 10. 六周范围警戒

首版只证明闭环：自由锻造连续刀坯 -> 隐藏性能与缺陷 -> 受控装配 -> NPC 自主冒险 -> 因果故事。

软体物理、全武器拓扑、任意焊接、运行时 LLM、多世界、正式微信版、经济成长和大量正式美术全部延期。新增需求只有直接增强核心闭环且不破坏当前切片验收时才进入本轮。
