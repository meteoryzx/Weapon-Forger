## A. 项目定位
- **A1 定位=求职作品集（游戏策划岗），非商业产品。** 理由：作者大三、腾讯 WXG 产品策划实习未转正、秋招需要作品。→ 重"设计深度 + 完整可玩核心闭环"，**不追求商业完整度/买量/长线**。
- **A2 目标平台=微信小游戏（远期）。** 理由：核心的"好友冒险故事分享"玩法为微信强社交设计。
- **A2b Demo 阶段导 H5 网页版验证，暂不碰微信开发者工具（本次决策 2026-07-16）。** 理由：作品集看的是玩法/系统/可玩性，面试官不在乎跑在哪；网页链接分享比"装微信工具才能看"友好得多。Cocos 支持一键导出 H5 与微信小游戏两个平台，故开发期只用 Cocos + 浏览器；社交分享 Demo 期用**网页链接平替**演示（逻辑同二维码）。**微信小游戏版留到真正上线时再接**（用户认为上线尚需很久，Demo 不考虑）。→ S0 验证标准从"微信里跑空白页"改为"Cocos 导出 H5、浏览器跑空白页"。
- **A3 引擎=Cocos Creator（TypeScript）。** 理由：既能导 H5（Demo 用）又能导微信小游戏（远期上线用），一举两得，保留了"随时变真小游戏"的能力。否决：① 手搓 Three.js 引擎 ② Unity（体积大、微信适配麻烦）③ 纯 H5（最快但以后上小游戏要重做，放弃了远期能力）。
- **A4 Demo 只做垂直切片**：教学(打第一把普通剑)→自由锻造→出6维武器→两出口(雇NPC自己看/二维码分享好友)→结局。**否决/推迟**：沙盒装饰购买、多NPC长线、大马士革全工艺、分级美术全库、真实服务端。理由：防范围膨胀，宁可小而完整。

## B. 开发方式
- **B1 Prompt 工程，作者不碰代码。** 作者角色=架构师+质检员。故一切要"可验证化"、模块要能独立验收。
- **B2 现阶段不使用多 Agent 团队。** 理由：单人单线、玩法未验证、探索期，多 Agent 管理成本 > 收益。触发时机=进入"批量生产 NPC 故事线/冒险节点"的内容爆发期再引入。否决：一上来就搭 Agent 团队。
- **B3 原型"验证即弃、不迁移"。** H5/探路原型只为验证结论，验证完从零用 Cocos 重写；**不做 H5→小游戏的迁移**（那是已知的大坑）。

## C. 锻造系统
- **C1 系统本质=工作台状态机，非固定流水线。** 玩家自由排列原子操作，材料带状态，操作有合法门槛。大马士革=敲薄→锯→叠→熔 循环 N 次，同一套操作不同排列，不动地基。
- **C2 五原子操作×三手势（点/拖/滑）。** 底层状态四类：形态/微观结构/成分/表面。
- **C3 锻打/打磨=参数化伪形变，不做真实软体物理仿真。** 锻打=刀坯分段改厚度/长度参数实时重绘；打磨=遮罩擦除。否决：真实物理形变（重引擎、一人做不了、无必要）。唯一难点=锻打手感调校（S5）。
- **C4 图纸=可无视的"新手拐杖"，非工艺锁。** 懂行者可无视图纸自由发挥。总目标=自由度够高 + 面向做武器优化。
- **C5 未记载工艺反馈=方案C**：自由发挥的效果如实反映到数值/外观（人人可得）；但"命名/图鉴"只给图纸记载工艺。否决：方案A(只认图纸工艺，打击探索)、方案B(工艺自动识别库，太重，预留未来)。
- **C6 工具=精进式（提升质量上限，不锁操作）。** 徒手能做任何操作，工具越好该工序评分/成功率上限越高。否决：工具=解锁新操作（锁自由度）。
- **C7 失败即内容**：材料/工具不够时高级工艺翻车出"裂纹"，失败变笑料与故事素材，而非单纯惩罚。
- **C8 锻造工序定稿（2026-07-17，基于真实工艺研究 docs/锻造工艺研究报告.md）**：真实13道工序抽象成游戏步骤：选料→加热→锻打→(正火)→淬火→**回火**→研磨→装配；折叠/大马士革为扩展。
  - **新增"回火"步骤**：真实世界"硬↔韧"取舍靠回火温度调（越高越韧越软）。游戏做成"拖温度条"，把取舍变成**玩家主动决策**。比"淬火自动决定"高级。
  - **研磨=第二塑形阶段**（作者直觉正确）：真实中"刀能不能切靠磨出的刃面斜角"，研磨能改轮廓/减薄/定刃角，是锋利↔耐用取舍的操作点，非只开刃。修正旧C3"打磨=纯遮罩擦除"。
  - **淬火水/油=有意义的选择**：水更硬更险(易裂)、油更稳。
  - **折叠/大马士革按真实**：设计成"外观/收藏加成+翻车风险(夹层/裂)"，**非纯性能buff**（现代钢折叠对性能无实质提升，主要花纹）。
- **C9 两条统领准则（2026-07-17）**：① **"拟真↔游戏"是可滑动旋钮非二选一**，平衡点原型迭代找（也是技术保险：做不动就往游戏端滑）；② **"能离散就不连续"**（旋转固定90°共4面=选面来敲），把操控地狱降成可控组合，几乎不损失深度。
- **C10 技术壁垒判断（2026-07-17）**：**全流程唯一真壁垒=锻打/研磨实时形变**，其余(回火/淬火/正火/翻车判定)全是纯逻辑=AI擅长=零壁垒。**理解加深不提高技术壁垒**（深度加在数值/因果，非形变技术）。形变三做法本质是"变形对象"不同(A分段/B顶点/C物理)非纯精度差；**用A分段缩放跨壁垒**(段够多≈平滑+被描边火花掩盖)，S5不够再局部借B，永不做C。**手感70%靠反馈(火花/震动/音效/数字跳动)，非形变精度**——S5调重点是反馈爽度。
- **C11 操控=双手协同+PC为主（2026-07-17）**：左手键盘(W/S送料,A/D转面90°)+右手鼠标(点击=敲,长按=蓄力,力道不同效果不同)。**核心逻辑与输入层解耦**，保留手机/微信触屏适配可能。判定=宽容的**固定平衡点**(时机条绿区,可迭代微调)，**非动态难度DDA**(纠正过一次误解:作者要的是"照顾挫败感的固定平衡点靠迭代调"，非按玩家实时表现补偿)。

## D. 数值体系（经多轮深挖，重点看）
- **D1 三层结构**：物理模拟层(4根基量:成分/微观结构/几何/内部完整性) → 6维评估压缩层 → 故事层。**6维不是最根基，是根基量的计算输出 + 面向故事的压缩接口。**
- **D2 6维（锋利/硬度/韧性/重量/外观/平衡），等级制0-10。** 作为"物理性能"评估层基本完备。**否决过**：4维精简（深度不够）、8维+特质标签（太复杂）、单开"特质标签"系统（改为内化进6维）。
- **D3 关键修正——三对物理取舍**：硬度↔韧性、锋利↔刃口耐用、轻↔结实。**否决了"6条独立能力值"的旧设计**（那会导致"除重量外越高越好、无取舍"，探路原型暴露了此漏洞）。平衡/外观不参与对立=综合工艺奖励。手艺=更精准分配+减损耗，非无脑全满。特殊工艺=缓解对立(如大马士革硬+3时韧只-0.5)。
- **D4 耐久=由硬度+韧性派生，内化，不单开第7维。** 保持简洁。（🔶 若后续发现耐久特别能出故事，再评估提为正式维度。）
- **D5 特殊功能（淬毒/引火/认主）属另一支**，走 process 工艺标签/故事层，不塞进物理6维。一把刀完整评价=物理性能支(6维)+特殊功能支(工艺标签)。
- **D6 锻造模拟愿景=Powder Toy/Noita 式属性涌现**：不写"包钢=+5"人为规则，而是模拟4根基量让优劣自动涌现。**放 Demo 之后**：Demo 期 M3 用简化取舍公式，验证好玩后再把 M3 内部升级为物理模拟，WeaponData 接口不变。需先收集真实工艺样本拆物理关系。
- **D7 总评分**：6维加权+工艺完成度，仅用于"决定套哪档品质贴图"+"社交门面数字"，**不参与冒险判定**（判定逐维判，保偏科翻车戏剧性）。
- **D8 数值显示=等级制**，玩家看0-10等级不看裸数字。

## E. 故事系统
- **E1 三层生成结构**：情境库 × 6维分段 × 叙事碎片库。加 DLC=塞情境/碎片，6维系统不改。
- **E2 故事"好看"四源泉**：反差/因果/代入/稀有。数值驱动前三之三，社交驱动代入。
- **E3 难度=结局稀有度，非通关难度。** 大部分冒险正常武器能过，稀有/隐藏结局设硬门槛。否决：把游戏做成"打不过关"的硬核难度。
- **E4 冒险判定=数据驱动规则表**（判定节点表），不硬编码。
- **E5 卖点二形态=轻跑团（2026-07-17）。** 不是"数值→固定判定分支"，而是**武器数值 + 玩家选择 + 世界状态变量 共同涌现故事**（像跑团）。
  - **防坑A(最优解)**：不同冒险线考验不同数值组合（潜行爱轻+锋利、攻坚爱重+硬度…），**无通用最优解**；"想去哪冒险"决定"造什么武器"，保护探索欲。
  - **防坑B(内容黑洞)**：用**"世界状态变量+条件文案"**实现轻跑团——选择改少量状态变量(名声/结仇/体力)，后续节点读"变量+武器数值"读出不同文案，**主干不真分叉成多条剧情**。参考《这是我的战争》。**否决"重跑团(剧情大分叉)"**——一人+秋招必烂尾。
  - **分支质量三原则**：单节点分段(非及格线,4档4剧情+情绪梯度) / 前节点结果影响后节点(因果连锁) / 反差节点(高数值反坏事)。
  - **⏳ 关键时序**：**卖点二完整设计推迟到锻造(S3)做完、有真实武器后再做**（脱离真武器设计跑团=空想，假数值必被真实分布推翻）。M4 届时从"纯判定"升级为"世界状态机+条件规则"，仍只读 WeaponData 6维（接口不变）。

## F. 社交
- **F1 双出口**：① 雇 NPC 冒险自己看（即时反馈，**解决"铁匠看不到故事高潮"的结构漏洞**）② 二维码分享好友看（传播）。否决：只做"分享给好友"单出口（会导致铁匠玩家没即时爽点、留存脆弱）。
- **F2 分享=二维码带6维参数，H5 现算故事**，不存整段故事。
- **F3 社交双引擎**：搞怪向(即时传播) + 收集向(隐藏结局钩子，冒险途中须"暗示"存在)。

## G. 美术
- **G1 基调=小丑牌(Balatro)式**：略带像素+相对精致+厚描边+鲜明暖色+讲究光影（看似像素实则精致）。**否决**：纯低保真像素（太糙）、纯扁平矢量（太轻）、真3D（一人扛不住量）、REPLACED式忧郁暗调（与轻松基调不符）。
- **G2 风险与退路**：AI 难一次产出小丑牌精致度，**先试做样例验证**；达不到就退回"扁平矢量+厚描边+暖色光影"。风格统一 > 单张精美。
- **G3 武器外观 = 物理外观两层模型（2026-07-16 重大修正）。** 玩家用锻造系统"打"出外观，**不是从预制部件库里选零件拼装**。
  - **第一层（系统实时生成）**：形状(网格变形)、温度色、打磨光泽——由锻造物理状态实时驱动，无需美术预制，保证"敲一下就变"。
  - **第二层（工艺涌现贴图）**：物理量达阈值时系统客观判定"某工艺达标"→叠加分级预制贴图(大马士革水纹/包钢分层线/镜面)，工艺越精湛套越高档。这是美术精致度(小丑牌质感)的来源，但由锻造涌现、非玩家选择。
  - **基础形制起手 + 锻造自由塑形**：玩家选刀/剑/斧基础形制起手（作为第二层贴图对齐的骨架锚点），锻造决定具体形状/厚薄/材质/数值。
  - **⚠️ 否决并废弃：早期"刀身/护手/柄拆图层槽、25素材拼1440把"的部件拼装库。** 否决理由：它是当初为"省美术工时"提的降本方案，却**悄悄把玩法从"锻造创造"降级成了"选零件拼装"**，与"玩家自己打出想要的武器"这一核心诉求冲突（一度造成美术与系统割裂）。两层模型用系统实时层替代了"拼装"，用工艺贴图层保住了精致度，缝合了割裂。
  - **意义**：美术资产清单 = 系统工艺种类 × 等级数（美术需求从系统长出来，不再拍脑袋）；美术是"锻造的结果展示"，与系统全然一体。花纹仍是 CS 淬火皮肤式分级思路，但触发权在锻造而非选择。
- **G4 美术落地放 S4 之后**，之前用占位图；"想法先入库，落地按顺序"。两层模型下：第一层(实时外观)随 S4 渲染模块出现即可见（占位色块也能跑）；第二层(工艺分级贴图)属精致美术，随 S7 内容接入阶段产出替换。

## I. 2026-07-20 六周执行方案（以下决策替代冲突旧项）

- **I1 最小物理内核进入首版 S3a，替代 D6 的“Demo 后再做”。** 这里的物理是确定性规则模型：温度、塑性、截面几何、近似体积、应力、完整性、淬火/回火状态；不是软体、有限元或逐粒子仿真。理由：锻造模拟与故事因果是作品的两个核心卖点，先用直接加减六维的公式会留下错误地基，也无法证明系统策划能力。
- **I2 模拟与评估拆开。** `ForgeState` 是施工真相；操作先改变材料状态；M3 只读最终状态并推导 `WeaponData`。这替代“Demo 期 M1/M3 共同写 WeaponData”的旧数据流。理由：让物理规则、视觉和故事可以分别测试和扩展。
- **I3 自由连续坯料不强制 `baseForm` 分类。** 首版从材料坯开始，8-10 个截面定义连续几何；怪异成品也必须合法进入故事。标准护手/握柄只在 S3d 受控装配时加入。未来多材料焊接通过 `WorkpieceState[] + JointState[]` 扩展，首版不开放任意焊接 UI。此项替代 G3 中“刀/剑/斧基础形制起手作为必要锚点”的部分。
- **I4 动态截面网格是首选，连接式 2D ribbon 是限时降级方案。** 二者读取同一 `ForgeSnapshot`，不改变模拟接口。继续否决软体物理。理由：先争取连续视觉效果，同时用两天技术限时保护六周范围。
- **I5 缺陷确定性产生。** 裂纹、弯曲和连接问题来自明确阈值与状态组合，不用随机概率制造。随机内容必须带显式种子；同一初态和操作序列结果一致。
- **I6 不使用万能总战力。** 旧 `overall` 改为 `craftsmanship`，只表示工艺完成度；用途由六维和结构化 `traits/flaws` 共同表达。六维对玩家隐藏，锻造界面不显示“优秀/失败”。NPC 自主冒险是性能的唯一明确反馈。
- **I7 故事由规则代理产生。** 首版一个世界、三类随机 NPC 委托；NPC 按目标、性格、体力、伤势、位置、关系、世界状态和武器能力行动。同一 `WeaponData` 与种子产生同一故事；不运行 LLM，不允许文案编造数据中不存在的事实。此项细化并替代旧“玩家选择为核心”的轻跑团表述，首版重点是 NPC 自主性。
- **I8 协作单位从“单模块”改为“可玩的纵向切片”。** 一个 2-4 天 PR 可以修改多个模块，但只能完成一条玩家可验证行为；禁止各模块长期分支最后集中拼装。所有修改经 Issue、Goal、短 branch、Draft PR、CI、作者试玩、Codex review 后 squash merge。
